# 个人 Markdown 笔记软件设计文档

## 1. 文档信息

- 项目名称：LinkNotes（暂定名）
- 文档类型：设计文档
- 文档版本：v0.1
- 编写日期：2026-03-23
- 关联文档：`docs/note-app-requirements.md`

## 2. 设计目标

本设计文档用于描述第一阶段 Windows 桌面端笔记软件的整体技术方案与模块划分。设计重点如下：

- 使用成熟技术快速落地可用版本
- 保持 Markdown 文件的开放性
- 确保快捷命令和原生 Markdown 共生
- 保持未来扩展 Android 和索引能力的可能性

## 3. 技术选型

### 3.1 总体选型

- 桌面壳：Electron
- 前端框架：React
- 开发语言：TypeScript
- 编辑器：CodeMirror 6
- Markdown 解析与预览：markdown-it
- 终端显示：xterm.js
- 终端后端：node-pty
- 构建工具：Electron Forge

### 3.2 选型理由

#### Electron

- 适合 Windows 桌面端快速开发
- 易于访问本地文件系统和系统能力
- 与终端、导出、菜单等桌面特性结合方便

#### React

- 适合搭建文件树、编辑器容器、预览面板和设置页
- 组件化便于维护三栏界面和底部终端面板

#### CodeMirror 6

- 非常适合 Markdown 编辑器
- 扩展机制成熟，适合实现快捷插入、补全和键位绑定
- 可在不改变文件格式的前提下增强编辑体验

#### markdown-it

- 解析 Markdown 成熟稳定
- 便于后续添加插件和扩展语法
- 渲染结果容易配合主题 CSS 控制视觉样式

#### xterm.js + node-pty

- 常见于桌面开发工具
- 可满足底部终端抽屉的主要需求

## 4. 总体架构

### 4.1 分层结构

- 主进程层：负责窗口、菜单、文件系统访问、终端会话、导出能力
- 预加载层：通过 `contextBridge` 暴露安全 API
- 渲染层：负责 UI、编辑器、预览、文件树和状态管理

### 4.2 模块图

- `main/`
  - Electron 主进程入口
  - 文件系统服务
  - 终端服务
  - 导出服务
- `preload/`
  - 对外暴露工作区、文件、终端、设置 API
- `renderer/`
  - 应用框架
  - 页面布局
  - 文件树模块
  - 编辑器模块
  - 预览模块
  - 终端模块
  - 搜索模块
  - 设置模块

## 6. UI 设计

### 6.1 主界面布局

- 左侧：工作区与文件树
- 中间：Markdown 编辑器
- 右侧：Markdown 预览
- 顶部：文件标题、搜索、视图切换、终端入口
- 底部：状态栏或抽屉式终端

### 6.2 布局原则

- 三栏是核心结构，终端作为第四块临时区域，不长期占据主布局
- 各栏宽度可调
- 小窗口下优先保留编辑区，预览区可折叠

### 6.3 视觉方向

- 浅色主题优先
- 使用大量留白和柔和边框
- 支持后续主题切换
- 主题样式应尽量通过 CSS 变量管理

## 7. 模块设计

### 7.1 工作区模块

职责：

- 打开工作区
- 记录最近工作区
- 提供工作区根路径
- 触发文件树刷新

关键接口建议：

- `openWorkspace()`
- `getRecentWorkspaces()`
- `getWorkspaceTree()`

### 7.2 文件树模块

职责：

- 渲染目录树
- 处理选中、展开、折叠
- 支持新建、重命名、删除、移动

关键状态：

- 当前选中文件
- 展开的目录集合
- 收藏和最近列表

### 7.3 编辑器模块

职责：

- 管理 CodeMirror 6 实例
- 提供内容编辑
- 处理快捷命令展开
- 处理自动保存

关键设计点：

- 编辑器内部状态与文件内容状态分离
- 对外暴露光标位置、文档内容、变更事件
- 自动保存应做节流或防抖

### 7.4 预览模块

职责：

- 使用 markdown-it 将文本渲染为 HTML
- 控制同步滚动
- 负责主题样式应用

关键设计点：

- 渲染逻辑与 UI 容器分离
- 后续支持目录锚点和代码高亮扩展

### 7.5 快捷命令模块

职责：

- 识别编辑器内的 `/命令`
- 执行模板替换
- 后续支持命令候选菜单

初始命令表：

- `/h1` -> `# `
- `/h2` -> `## `
- `/h3` -> `### `
- `/dmk` -> 代码块模板
- `/code` -> 代码块模板
- `/todo` -> `- [ ] `
- `/quote` -> `> `
- `/link` -> 链接模板
- `/img` -> 图片模板
- `/table` -> 表格模板

关键原则：

- 只替换输入内容，不引入私有 Markdown 语法
- 文件最终保存内容始终为标准 Markdown

### 7.6 搜索模块

职责：

- 搜索文件名
- 搜索文件内容
- 呈现匹配结果和定位信息

第一阶段实现建议：

- 初期可直接遍历 Markdown 文件进行搜索
- 后续可引入 SQLite 索引提升性能

### 7.7 终端模块

职责：

- 管理底部终端抽屉
- 建立前端与 pty 的通信
- 设置工作目录

关键设计点：

- 一个终端会话对应一个 pty 实例
- 默认使用当前工作区或当前笔记目录作为 cwd
- 仅在桌面端启用

### 7.8 导出模块

职责：

- 导出当前文档为 HTML
- 导出当前文档为 PDF

实现建议：

- HTML 导出基于 markdown-it 输出
- PDF 导出可通过 Electron 的打印或页面导出能力实现

## 8. 数据设计

### 8.1 文件组织

建议使用文件系统作为主数据源：

- 笔记：`.md`
- 附件：工作区内独立 `assets/` 或 `attachments/` 目录
- 配置：应用本地配置文件

示例：

```text
workspace/
  notes/
    今日整理.md
    读书摘录.md
  attachments/
    image-001.png
```

### 8.2 应用配置

应用配置不应混入用户笔记内容，可保存在系统用户目录，例如：

- 最近工作区
- 窗口布局
- 主题配置
- 默认终端类型
- 快捷命令配置

### 8.3 后续索引设计

如果需要提升搜索和统计能力，可引入 SQLite 作为索引层，而不是主存储层。

索引层建议记录：

- 文件路径
- 标题
- 修改时间
- 正文摘要
- 标签信息

## 9. 关键交互设计

### 9.1 文件打开流程

1. 用户在文件树中点击文件
2. 读取文件内容
3. 更新编辑器内容
4. 更新预览内容
5. 更新状态栏和当前路径

### 9.2 自动保存流程

1. 用户修改编辑器内容
2. 编辑器抛出变更事件
3. 前端更新内存状态
4. 防抖后调用保存接口
5. 主进程写入文件
6. 返回保存状态

### 9.3 快捷命令展开流程

1. 用户输入 `/h2`
2. 用户按 `Tab`
3. 编辑器插件检测到命令
4. 将 `/h2` 替换为 `## `
5. 光标定位在标题输入位置

### 9.4 图片拖拽流程

1. 用户拖入图片到编辑器
2. 前端接收文件对象
3. 主进程将文件复制到附件目录
4. 返回相对路径
5. 编辑器插入 Markdown 图片语法

### 9.5 终端打开流程

1. 用户点击终端按钮或按快捷键
2. UI 展开底部终端面板
3. 创建或激活终端会话
4. 终端 cwd 指向当前工作区或当前笔记目录

## 10. IPC 与安全设计

### 10.1 安全原则

- 启用 `contextIsolation`
- 不在渲染进程直接暴露 Node 全能力
- 所有文件和终端能力均通过预加载层白名单 API 暴露

### 10.2 IPC 主题建议

- `workspace:*`
- `file:*`
- `search:*`
- `terminal:*`
- `export:*`
- `settings:*`

### 10.3 示例接口

- `workspace:open`
- `workspace:getTree`
- `file:read`
- `file:save`
- `file:create`
- `file:rename`
- `file:delete`
- `search:query`
- `terminal:create`
- `terminal:write`
- `terminal:resize`
- `terminal:kill`
- `export:html`
- `export:pdf`

## 11. 状态管理建议

建议按模块管理前端状态：

- 工作区状态
- 当前文件状态
- 编辑器状态
- 预览状态
- 终端状态
- 设置状态

实现上可先采用轻量方式，如 React Context 或 Zustand，避免过早引入复杂架构。

## 12. 主题与样式设计

### 12.1 主题策略

- 使用 CSS 变量管理色彩、边框、阴影、字体
- 编辑器主题与预览主题解耦，但共享基础设计 token

### 12.2 Typora 风格借鉴原则

- 可以借鉴 Typora 风格的排版和配色
- 不直接依赖 Typora 的 DOM 结构
- 将样式抽象为自己的预览容器主题

### 13.1 第二阶段

- 同步能力对接

## 14. 风险与取舍

### 14.1 主要风险

- 大工作区下搜索性能可能不足
- 同步滚动实现不当会影响体验
- 终端能力需要注意安全边界

### 14.2 当前取舍

- 第一阶段使用文件系统作为主存储，避免提前引入数据库复杂度
- 第一阶段不做复杂命令面板，先做快捷命令展开
- 第一阶段不做 Android 编辑端，先聚焦桌面体验
